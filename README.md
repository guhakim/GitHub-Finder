> **"AI가 코드를 만들어도, 최종 판단은 개발자가 한다."**  
> AI와 협업하되, 코드의 흐름을 직접 이해하고 검증하는 것이 핵심입니다.

---

## 프로젝트 소개

**GitHub Finder**는 GitHub 사용자명을 검색하면 해당 사용자의 프로필 정보와 최신 저장소 목록을 실시간으로 보여주는 웹 앱입니다.

단순한 화면 구성이 아닌, **GitHub REST API와 직접 통신하여 실제 데이터를 받아오고 화면에 렌더링**하는 것에 초점을 맞췄습니다.

---

## 학습 목표

- GitHub REST API 엔드포인트 구조를 이해한다
- `fetch()`와 `async/await`를 사용한 비동기 데이터 요청을 구현한다
- API 응답 JSON에서 필요한 필드를 추출해 DOM에 렌더링한다
- 빈 입력, 존재하지 않는 사용자(404), 네트워크 오류 등 예외 상황을 처리한다
- AI 도구를 활용하되, 결과물을 직접 검토하고 수정하는 협업 흐름을 익힌다

## 주요 기능

### 사용자 검색
- 검색창에 GitHub 사용자명 입력 후 **검색 버튼** 또는 **Enter 키**로 검색
- 빈 입력값 제출 시 안내 메시지 표시

### 프로필 카드 출력
- 프로필 이미지, 이름, 사용자명(`@login`)
- 소개글(bio)
- 팔로워 / 팔로잉 / 공개 저장소 수
- GitHub 프로필 외부 링크

### 최신 저장소 목록
- 최근 업데이트 기준 최신 5개 저장소 표시
- 저장소명 (클릭 시 GitHub로 이동), 설명, 스타 수, 주요 언어

### 상태 메시지 처리

| 상황 | 메시지 |
|------|--------|
| 검색 중 | "불러오는 중..." |
| 빈 입력값 | "사용자명을 입력해 주세요." |
| 존재하지 않는 사용자 | `"username" 사용자를 찾을 수 없습니다.` |
| 네트워크 오류 | "네트워크 오류가 발생했습니다. 인터넷 연결을 확인해 주세요." |
| API 오류 (403 등) | "요청 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |

## 구현 흐름도

```
[입력] 사용자명 입력 + Enter / 버튼 클릭
    ↓
[유효성 검사] 빈 값이면 에러 메시지 출력 후 중단
    ↓
[API 요청] GitHub API에 사용자 정보 + 저장소 목록 동시 요청
    https://api.github.com/users/{username}
    https://api.github.com/users/{username}/repos?sort=updated&per_page=5
    ↓
[응답 처리] 404 → 에러 메시지 / 정상 → JSON 파싱
    ↓
[렌더링] 프로필 카드 + 저장소 목록 DOM 출력
```

## AI 생성 결과 검토 기준

AI가 생성한 코드를 그대로 사용하지 않고, 아래 기준으로 직접 검토했습니다.

| 점검 항목 | 확인 내용 |
|-----------|-----------|
| **통신 점검** | API 요청 URL 파라미터가 올바른가, JSON 응답이 정상 수신되는가 |
| **렌더링 점검** | 프로필 데이터가 올바른 DOM 요소에 매핑되는가, 저장소 목록이 반복 출력되는가 |
| **안정성 점검** | 빈 입력값 차단 로직이 있는가, 404 오류가 UI를 깨지 않는가 |
| **기록 점검** | 브라우저 Console에 빨간 오류(Red log)가 없는가 |

## 수정 요청 내용

AI가 처음 생성한 결과에서 직접 수정하거나 요청한 내용입니다.

### 수정 1 - `bio` 필드 null 처리
- **현상:** `bio`가 null인 사용자 검색 시 "null" 텍스트가 화면에 출력됨
- **원인:** `user.bio`를 그대로 `textContent`에 넣어서 null이 문자열로 변환됨
- **해결:**
```javascript
// 수정 전
document.getElementById('bio').textContent = user.bio;

// 수정 후
document.getElementById('bio').textContent = user.bio || '';
```

### 수정 2 - `name` 필드 없는 사용자 처리
- **현상:** `name`을 설정하지 않은 계정에서 빈 이름이 표시됨
- **해결:** `user.name || user.login`으로 이름이 없을 때 사용자명으로 대체

## 배운 점

1. **비동기 처리의 구조** - `async/await`가 Promise를 어떻게 감싸는지, `Promise.all()`로 병렬 요청이 가능한 이유를 이해했습니다.

2. **API 응답은 항상 검증해야 한다** - `ok` 속성과 `status` 코드로 성공/실패를 명시적으로 구분하지 않으면, 오류 응답도 정상처럼 처리될 수 있습니다.

3. **null/undefined는 항상 대비해야 한다** - API 응답 필드는 값이 없을 때 `null`을 반환합니다. `|| ''` 또는 `|| '기본값'`으로 방어 코드를 작성하는 습관이 중요합니다.

4. **AI 코드는 검토가 필수다** - AI가 생성한 코드는 문법적으로는 맞지만, 실제 데이터에서 발생하는 edge case를 놓치는 경우가 있습니다. 직접 테스트하고 수정하는 과정이 진짜 학습입니다.

## 3줄 보고서

1. GitHub REST API에서 사용자 정보와 저장소 목록을 비동기로 요청하고, 응답 JSON을 DOM에 렌더링하는 전체 흐름을 구현했습니다.
2. 빈 입력, 404 없는 사용자, 네트워크 오류 등 세 가지 예외 상황을 각각 다른 메시지로 안내하는 예외 처리를 완성했습니다.
3. AI가 생성한 코드를 그대로 쓰지 않고, null 필드 처리 등 실제 데이터에서 발생하는 문제를 직접 발견하고 수정하는 검토 과정을 경험했습니다.
